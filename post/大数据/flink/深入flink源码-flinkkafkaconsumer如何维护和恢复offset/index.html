从TopicPartitionState包含的字段可以看出，消费者自己维护了最近消费的offset（position字段）和最近提交的offset（committed字段），当消费者向Kafka server提交最近消费的offset时，就会更新committed字段的值，而Kafka server就会更新消费者最近的消费情况（根据版本不同，可能保存在一个topic或zookeeper中）。关于KafkaConsumer的原理和消费的整个流程，会在分析Kafka源码的文章中详细说明。这里只需要了解，Flink消费端集成了KafkaConsumer，那么显而易见也会维护消费者的消费情况。

那么问题来了。当开启了Flink的checkpoint机制时，Flink是从Kafka server或zookeeper去取之前保存的offset吗？整个消费过程中对offset的维护是遵从KafkaConsumer的机制还是Flink自己有一套机制？如果把这个问题搞清楚，不但可以明白FlinkKafkaConsumer如何维护offset，还可以对Flink checkpoint机制有更深的理解，甚至可以为其他任何一个框架扩展消费Kafka数据的connector。

public class CheckpointCoordinatorDeActivator implements JobStatusListener {

   private final CheckpointCoordinator coordinator;

   public CheckpointCoordinatorDeActivator(CheckpointCoordinator coordinator) {
      this.coordinator = checkNotNull(coordinator);
   }

   @Override
   public void jobStatusChanges(JobID jobId, JobStatus newJobStatus, long timestamp, Throwable error) {
      if (newJobStatus == JobStatus.RUNNING) {
         // start the checkpoint scheduler
         coordinator.startCheckpointScheduler();
      } else {
         // anything else should stop the trigger for now
         coordinator.stopCheckpointScheduler();
      }
   }
}

public void startCheckpointScheduler() {
		synchronized (lock) {
			if (shutdown) {
				throw new IllegalArgumentException("Checkpoint coordinator is shut down");
			}

			// make sure all prior timers are cancelled
			stopCheckpointScheduler();

			periodicScheduling = true;
			currentPeriodicTrigger = scheduleTriggerWithDelay(getRandomInitDelay());
		}
}

private ScheduledFuture<?> scheduleTriggerWithDelay(long initDelay) {
		return timer.scheduleAtFixedRate(
			new ScheduledTrigger(),
			initDelay, baseInterval, TimeUnit.MILLISECONDS);
}

@Override
public CompletableFuture<Acknowledge> triggerCheckpoint(
      ExecutionAttemptID executionAttemptID,
      long checkpointId,
      long checkpointTimestamp,
      CheckpointOptions checkpointOptions,
      boolean advanceToEndOfEventTime) {
   log.debug("Trigger checkpoint {}@{} for {}.", checkpointId, checkpointTimestamp, executionAttemptID);

   final CheckpointType checkpointType = checkpointOptions.getCheckpointType();
   if (advanceToEndOfEventTime && !(checkpointType.isSynchronous() && checkpointType.isSavepoint())) {
      throw new IllegalArgumentException("Only synchronous savepoints are allowed to advance the watermark to MAX.");
   }
	 // 1.根据参数取出Task。
   final Task task = taskSlotTable.getTask(executionAttemptID);

   if (task != null) {
      // 2.触发对应Task的checkpoint过程。
      task.triggerCheckpointBarrier(checkpointId, checkpointTimestamp, checkpointOptions, advanceToEndOfEventTime);

      return CompletableFuture.completedFuture(Acknowledge.get());
   } else {
      final String message = "TaskManager received a checkpoint request for unknown task " + executionAttemptID + '.';

      log.debug(message);
      return FutureUtils.completedExceptionally(new CheckpointException(message, CheckpointFailureReason.TASK_CHECKPOINT_FAILURE));
   }
}

public void produce(final ConsumerRecords<byte[], byte[]> element)
      throws InterruptedException, WakeupException, ClosedException {

   checkNotNull(element);

   synchronized (lock) {
      while (next != null && !wakeupProducer) {
         lock.wait();
      }

      wakeupProducer = false;

      if (next != null) {
         throw new WakeupException();
      }

      else if (error == null) {
         // 将最新拉取的数据赋给Handover对象的next字段。
         next = element;
         lock.notifyAll();
      }

      else {
         throw new ClosedException();
      }
   }
}

public ConsumerRecords<byte[], byte[]> pollNext() throws Exception {
   synchronized (lock) {
      while (next == null && error == null) {
         lock.wait();
      }

      // 将next字段的值符给 n 后返回 n。
      ConsumerRecords<byte[], byte[]> n = next;
      if (n != null) {
         next = null;
         lock.notifyAll();
         return n;
      }
      else {
         ExceptionUtils.rethrowException(error, error.getMessage());

         // this statement cannot be reached since the above method always throws an exception
         // this is only here to silence the compiler and any warnings
         return ConsumerRecords.empty();
      }
   }

            break;
         default:
            for (KafkaTopicPartition seedPartition : allPartitions) {
               subscribedPartitionsToStartOffsets.put(seedPartition, startupMode.getStateSentinel());
            }
      }
     
     ......

   }
}

还可以看到，如果不是从checkpoint或savepoint恢复，则按照配置的EARLIEST、LATEST等方式找到offset，赋给subscribedPartitionsToStartOffsets。随后，subscribedPartitionsToStartOffsets又把值赋给了fetcher中的subscribedPartitionStates字段。

pendingOffsetsToCommit

在snapshotState方法中，还有一个重要字段之前没有讲，即pendingOffsetsToCommit。该方法中间还有这样的逻辑。

if (offsetCommitMode == OffsetCommitMode.ON_CHECKPOINTS) {
   pendingOffsetsToCommit.put(context.getCheckpointId(), restoredState);

这个OffsetCommitMode，就是指的Flink将offset提交到Kafka的策略。ON_CHECKPOINTS策略的意思是会在checkpoint成功时主动commit。

它是从nextOffsetsToCommit字段中取到的值。说明在某个时刻，pendingOffsetsToCommit中的值赋给了nextOffsetsToCommit。

这个过程发生在notifyCheckpointComplete方法中。FlinkKafkaConsumerBase实现了CheckpointListener接口，所以拥有了这个方法。这个方法调用fetcher.commitInternalOffsetsToKafka方法，进而调用了consumerThread.setOffsetsToCommit，将值赋给了nextOffsetsToCommit字段。

notifyCheckpointComplete方法会在checkpoint触发后调用。具体调用链比较长，触发的起点可追溯到StreamTask的executeCheckpointing方法中。该方法在执行完checkpointStreamOperator操作后，会实例化一个AsyncCheckpointRunnable异步任务，该任务所做的事情就是通知checkpoint完成，并触发算子的notifyCheckpointComplete方法。

小结

至此FlinkKafkaConsumer关于offset的操作已经从头到尾理了一遍。可以得出的结论是，Flink自己维护了一套offset，在恢复时直接从checkpont或savepoint文件中取到之前保存的状态，以此为起始点告诉KafkaConsumer开始消费。

还有个遗留的小问题，就是在做snapshotState时，说到过要先取出所有已注册的状态，然后将其写入文件。但没有说这些状态是何时注册的。这些状态是在initialize方法中，调用stateStore.getUnionListState时注册的。而initialize方法和上面说到的open方法，会在StreamTask端做初始化时依次调用。
