Flink关于并行度有两个概念——并行度与最大并行度。在做业务代码的开发时，算子后面可以跟setMaxParallelism或者setParallelism，在配置中也可对其进行设置。

这两者在Flink状态管理和恢复过程中起着重要作用。

为什么要有最大并行度

最大并行度的概念主要是针对KeyedState提出的。通过对Checkpoint写入和读取流程的分析，我们可以设想一下，在改变了算子并行度的情况下，让我们自己实现KeyedState的恢复，应该如何做？

一般逻辑是像上图一样。设原来有3个并行度，经过keyBy操作后，key为0、3、6……的数据到了Subtask 0中，key为1、4、7……的数据到了Subtask 1中，key为2、5、8……的数据到了Subtask 2中。在checkpoint阶段，每个并行实例独立进行状态数据的存储，于是，数据写入到文件后如上图排列。

当并行度改为4后重启，这时涉及到状态数据的分配问题。能不能直接将0、3、6……这些数据放入新的Subtask 0，将1、4、7……放入Subtask 1，将2、5、8……放入Subtask 2，然后Subtask 3中没有数据呢？显然是不行的。因为之后当有新的数据到来时，会按照新的并行度来选择进入到哪个并行实例。算法如上图底部所示，先对key取hash，然后度并行度取余得到并行实例的索引。这样的分配是没有意义的。

显然，正确的做法是，对状态数据文件中的每个key计算hash并取余，算出它应该被加载到哪个并行实例中。一般在真实场景中，key的量是巨大的，这个计算过程比较慢；另外一个更糟糕的结果，如上图所示，是当新的并行实例去拉取数据时，会出现非常多的随机IO，并且请求数也是巨大的，网络IO效率非常低。

为了对此过程进行优化，Flink提出了最大并行度的概念。最大并行度的个数表示有多少个KeyGroup。如下图所示：

比如有10个KeyGroup。有了这个概念，整个过程就变得非常不一样。首先，数据不是按照key去决定它被分配到哪个Subtask。它会先根据key决定它属于哪个KeyGroup。key为1、11……的数据属于KeyGroup 1，key为2、12……的数据属于KeyGroup 2，以此类推。然后，会根据最大并行度和并行度计算出每个Subtask负责哪部分KeyGroup。当并行度为3时，KeyGroup 0～KeyGroup 3由Subtask 0负责，KeyGroup 4~KeyGroup 6由Subtask 1负责，KeyGroup 7~KeyGroup 9由Subtask 2负责。这样，每条数据就知道自己应该分配到哪个并行实例。在checkpoint写入时，仍然时每个并行实例单独写入，在写入时，每个KeyGroup的数据排列在一起，如上图所示。

当并行度改变时，不需要计算每个key要被分配到哪个并行实例，而只计算每个KeyGroup应该被分配到哪里。并行度变成4时，KeyGroup 0~KeyGroup 2由新的Subtask 0负责，KeyGroup 3~KeyGroup4由Subtask 1负责，KeyGroup 5~KeyGroup 7由Subtask 2负责，剩下的由Subtask 3负责。因此，在读取时，少了大量随机IO，变成了顺序IO。Subtask 0只需要发一次请求就获得了所有数据，而Subtask 1的数据在两个文件中，也只需要发送两次请求就获得了所有数据。

这是因为，数据是根据key的hash值决定它属于哪个KeyGroup的。如果最大并行度发生改变（且设并行度未发生改变这种简单场景），那么恢复后新到来的数据很可能与之前相同key的数据不属于同一个KeyGroup，而由于最大并行度发生了改变，每个并行实例所负责的KeyGroup也会发生改变，后果是新来的数据与之前相同key的数据会在不同的Subtask中。这就会造成业务上的错误。

private Tuple2<List<KeyedStateHandle>, List<KeyedStateHandle>> reAssignSubKeyedStates(
      OperatorState operatorState,
      List<KeyGroupRange> keyGroupPartitions,
      int subTaskIndex,
      int newParallelism,
      int oldParallelism) {

   List<KeyedStateHandle> subManagedKeyedState;
   List<KeyedStateHandle> subRawKeyedState;

   if (newParallelism == oldParallelism) {
      if (operatorState.getState(subTaskIndex) != null) {
         subManagedKeyedState = operatorState.getState(subTaskIndex).getManagedKeyedState().asList();
         subRawKeyedState = operatorState.getState(subTaskIndex).getRawKeyedState().asList();
      } else {
         subManagedKeyedState = Collections.emptyList();
         subRawKeyedState = Collections.emptyList();
      }
   } else {
      subManagedKeyedState = getManagedKeyedStateHandles(operatorState, keyGroupPartitions.get(subTaskIndex));
      subRawKeyedState = getRawKeyedStateHandles(operatorState, keyGroupPartitions.get(subTaskIndex));
   }

   if (subManagedKeyedState.isEmpty() && subRawKeyedState.isEmpty()) {
      return new Tuple2<>(Collections.emptyList(), Collections.emptyList());
   } else {
      return new Tuple2<>(subManagedKeyedState, subRawKeyedState);
   }
}

public static List<KeyedStateHandle> getManagedKeyedStateHandles(
   OperatorState operatorState,
   KeyGroupRange subtaskKeyGroupRange) {

   final int parallelism = operatorState.getParallelism();

   List<KeyedStateHandle> subtaskKeyedStateHandles = null;

   for (int i = 0; i < parallelism; i++) {
      if (operatorState.getState(i) != null) {

         Collection<KeyedStateHandle> keyedStateHandles = operatorState.getState(i).getManagedKeyedState();

         if (subtaskKeyedStateHandles == null) {
            subtaskKeyedStateHandles = new ArrayList<>(parallelism * keyedStateHandles.size());
         }

         extractIntersectingState(
            keyedStateHandles,
            subtaskKeyGroupRange,
            subtaskKeyedStateHandles);
      }
   }

   return subtaskKeyedStateHandles;
}

先获取旧的并行度，然后对旧的并行度进行循环。每循环一次，就把以前一个并行实例负责的文件元数据取出来，看其中的KeyGroup范围有没有与新的并行实例负责的部分重合，如果有的话就把该handle对象赋给新的并行实例。主要思想就是寻找交叉的部分，在extractIntersectingState方法中。

private static void extractIntersectingState(
   Collection<KeyedStateHandle> originalSubtaskStateHandles,
   KeyGroupRange rangeToExtract,
   List<KeyedStateHandle> extractedStateCollector) {

   for (KeyedStateHandle keyedStateHandle : originalSubtaskStateHandles) {

      if (keyedStateHandle != null) {

         KeyedStateHandle intersectedKeyedStateHandle = keyedStateHandle.getIntersection(rangeToExtract);

         if (intersectedKeyedStateHandle != null) {
            extractedStateCollector.add(intersectedKeyedStateHandle);
         }
      }
   }
}

比如要给新的并行实例中的第二个并行实例赋值时，它负责的KeyGroup范围是4~6。进入到这个方法中rangeToExtract表示的就是4~6。此时originalSubtaskStateHandles表示的是旧并行实例中的第一个，包含0~4的元信息。因此keyedStateHandle.getIntersection方法就会将旧并行实例中KeyGroup 4的信息取出来构造成一个新的KeyedStateHandle对象。

就这样经过多层循环，给每个新的并行实例找到交叉部分并交由新的并行实例维护。

这样，新的并行实例就知道自己应该去哪些文件取数据，并且知道自己要取的KeyGroup在文件的什么位置。

   Map<String, StateTable<K, ?, ?>> registeredKVStates = new HashMap<>();
   Map<String, HeapPriorityQueueSnapshotRestoreWrapper> registeredPQStates = new HashMap<>();
   CloseableRegistry cancelStreamRegistryForBackend = new CloseableRegistry();
   HeapSnapshotStrategy<K> snapshotStrategy = initSnapshotStrategy(
      asynchronousSnapshots, registeredKVStates, registeredPQStates, cancelStreamRegistryForBackend);
   InternalKeyContext<K> keyContext = new InternalKeyContextImpl<>(
      keyGroupRange,
      numberOfKeyGroups
   );
   HeapRestoreOperation<K> restoreOperation = new HeapRestoreOperation<>(
      restoreStateHandles,
      keySerializerProvider,
      userCodeClassLoader,
      registeredKVStates,
      registeredPQStates,
      cancelStreamRegistry,
      priorityQueueSetFactory,
      keyGroupRange,
      numberOfKeyGroups,
      snapshotStrategy,
      keyContext);
   try {
      restoreOperation.restore();
   } catch (Exception e) {
      throw new BackendBuildingException("Failed when trying to restore heap backend", e);
   }
   return new HeapKeyedStateBackend<>(
      kvStateRegistry,
      keySerializerProvider.currentSchemaSerializer(),
      userCodeClassLoader,
      executionConfig,
      ttlTimeProvider,
      cancelStreamRegistryForBackend,
      keyGroupCompressionDecorator,
      registeredKVStates,
      registeredPQStates,
      localRecoveryConfig,
      priorityQueueSetFactory,
      snapshotStrategy,
      keyContext);

private void readStateHandleStateData(
   FSDataInputStream fsDataInputStream,
   DataInputViewStreamWrapper inView,
   KeyGroupRangeOffsets keyGroupOffsets,
   Map<Integer, StateMetaInfoSnapshot> kvStatesById,
   int numStates,
   int readVersion,
   boolean isCompressed) throws IOException {

   final StreamCompressionDecorator streamCompressionDecorator = isCompressed ?
      SnappyStreamCompressionDecorator.INSTANCE : UncompressedStreamCompressionDecorator.INSTANCE;

   for (Tuple2<Integer, Long> groupOffset : keyGroupOffsets) {
      int keyGroupIndex = groupOffset.f0;
      long offset = groupOffset.f1;

      Preconditions.checkState(keyGroupRange.contains(keyGroupIndex), "The key group must belong to the backend.");

      fsDataInputStream.seek(offset);

      int writtenKeyGroupIndex = inView.readInt();
      Preconditions.checkState(writtenKeyGroupIndex == keyGroupIndex,
         "Unexpected key-group in restore.");

      try (InputStream kgCompressionInStream =
             streamCompressionDecorator.decorateWithCompression(fsDataInputStream)) {

         readKeyGroupStateData(
            kgCompressionInStream,
            kvStatesById,
            keyGroupIndex,
            numStates,
            readVersion);
      }
   }
}

private void readKeyGroupStateData(
   InputStream inputStream,
   Map<Integer, StateMetaInfoSnapshot> kvStatesById,
   int keyGroupIndex,
   int numStates,
   int readVersion) throws IOException {

   DataInputViewStreamWrapper inView =
      new DataInputViewStreamWrapper(inputStream);

   for (int i = 0; i < numStates; i++) {

      final int kvStateId = inView.readShort();
      final StateMetaInfoSnapshot stateMetaInfoSnapshot = kvStatesById.get(kvStateId);
      final StateSnapshotRestore registeredState;

      switch (stateMetaInfoSnapshot.getBackendStateType()) {
         case KEY_VALUE:
            registeredState = registeredKVStates.get(stateMetaInfoSnapshot.getName());
            break;
         case PRIORITY_QUEUE:
            registeredState = registeredPQStates.get(stateMetaInfoSnapshot.getName());
            break;
         default:
            throw new IllegalStateException("Unexpected state type: " +
               stateMetaInfoSnapshot.getBackendStateType() + ".");
      }

      StateSnapshotKeyGroupReader keyGroupReader = registeredState.keyGroupReader(readVersion);
      keyGroupReader.readMappingsInKeyGroup(inView, keyGroupIndex);
   }
}

public void readMappingsInKeyGroup(@Nonnull DataInputView in, @Nonnegative int keyGroupId) throws IOException {
   int numElements = in.readInt();
   for (int i = 0; i < numElements; i++) {
      T element = readerFunction.readElement(in);
      elementConsumer.consume(element, keyGroupId);
   }
}

private static <K, N, S> StateSnapshotKeyGroupReader createV2PlusReader(
   StateTable<K, N, S> stateTable) {
   final TypeSerializer<N> namespaceSerializer = stateTable.getNamespaceSerializer();
   final TypeSerializer<S> stateSerializer = stateTable.getStateSerializer();
   final TypeSerializer<K> keySerializer = stateTable.keySerializer;
   final Tuple3<N, K, S> buffer = new Tuple3<>();
   return KeyGroupPartitioner.createKeyGroupPartitionReader((in) -> {
      buffer.f0 = namespaceSerializer.deserialize(in);
      buffer.f1 = keySerializer.deserialize(in);
      buffer.f2 = stateSerializer.deserialize(in);
      return buffer;
   }, (element, keyGroupId1) -> stateTable.put(element.f1, keyGroupId1, element.f0, element.f2));
}

调用elementConsumer.consume，就会调用到(element, keyGroupId1) -> stateTable.put(element.f1, keyGroupId1, element.f0, element.f2))。数据也就存入了stateTable中。

总结

本文分析了并行度与最大并行度之间的关系，并且演示了在修改并行度后，KeyedState是如何重新分配到各个算子并行实例上的。

在KeyedState进行保存的过程中，也是先写入状态元信息（序列化器等信息），然后按照KeyGroup依次写入。每个KeyGroup又有一个或多个状态，依次序列化到文件，记录每个KeyGroup在文件中的起始位置。
