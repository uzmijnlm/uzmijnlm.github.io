public void putLong(final long value) {
    try {
        appendStream.writeLong(value);
    } catch (IOException e) {
        throw new KafkaException("I/O exception when writing to the append stream, closing", e);
    }

public final void writeLong(long v) throws IOException {
    writeBuffer[0] = (byte)(v >>> 56);
    writeBuffer[1] = (byte)(v >>> 48);
    writeBuffer[2] = (byte)(v >>> 40);
    writeBuffer[3] = (byte)(v >>> 32);
    writeBuffer[4] = (byte)(v >>> 24);
    writeBuffer[5] = (byte)(v >>> 16);
    writeBuffer[6] = (byte)(v >>>  8);
    writeBuffer[7] = (byte)(v >>>  0);
    out.write(writeBuffer, 0, 8);
    incCount(8);
}

从这里就可以看出，Kafka的消息不仅包含消息本身的内容，还包含了一些消息自身的描述信息，如上面append方法中还出现了offset和size等信息的写入。其整体按照一定格式组合起来形成一条完整的消息。

对于一个消息中间件（其实不仅限于消息中间件）来说，约定好的消息格式可以使框架在整体上能够优化其占用空间、解析速度、IO速度等。本文将以Kafka 0.10为例梳理其消息的格式。

public Compressor(ByteBuffer buffer, CompressionType type) {
    this.type = type;
    this.initPos = buffer.position();

    this.numRecords = 0;
    this.writtenUncompressed = 0;
    this.compressionRate = 1;
    this.maxTimestamp = Record.NO_TIMESTAMP;

    if (type != CompressionType.NONE) {
        // for compressed records, leave space for the header and the shallow message metadata
        // and move the starting position to the value payload offset
        buffer.position(initPos + Records.LOG_OVERHEAD + Record.RECORD_OVERHEAD);
    }

    // create the stream
    bufferStream = new ByteBufferOutputStream(buffer);
    appendStream = wrapForOutput(bufferStream, type, COMPRESSION_DEFAULT_BUFFER_SIZE);
}

其中有一个关键步骤，就是判断Compressor的类型。如果不为NONE，会调用buffer.position方法移动position的位置。显然如果为NONE则不做任何操作。稍后会分析不为NONE时为何会移动position的位置。

public static int recordSize(byte[] key, byte[] value) {
    return recordSize(key == null ? 0 : key.length, value == null ? 0 : value.length);
}

public static int recordSize(int keySize, int valueSize) {
    return CRC_LENGTH + MAGIC_LENGTH + ATTRIBUTE_LENGTH + TIMESTAMP_LENGTH + KEY_SIZE_LENGTH + keySize + VALUE_SIZE_LENGTH + valueSize;
}

这意味着之后真正将消息写入buffer时一定也会将该消息的crc、magic_value、attribute、timestamp、key_size、value_size这些信息一同写入。那么这些信息是什么意思呢？

我们先继续分析流程。计算出消息大小后，调用compressor.putLong(offset)方法用8个字节存储offset的值。再次强调这里的offset是一个RecordBatch内部从0开始自增的一个偏移量，可以理解为消息在一个RecordBatch中的相对偏移量。接着，调用compressor.putInt(size)方法用4个字节存储消息的大小。然后调用putRecord方法将消息写入：

public long putRecord(long timestamp, byte[] key, byte[] value, CompressionType type,
                      int valueOffset, int valueSize) {
    long crc = Record.computeChecksum(timestamp, key, value, type, valueOffset, valueSize);
    byte attributes = Record.computeAttributes(type);
    putRecord(crc, attributes, timestamp, key, value, valueOffset, valueSize);
    return crc;
}

private void putRecord(final long crc, final byte attributes, final long timestamp, final byte[] key, final byte[] value, final int valueOffset, final int valueSize) {
    maxTimestamp = Math.max(maxTimestamp, timestamp);
    Record.write(this, crc, attributes, timestamp, key, value, valueOffset, valueSize);
}

public static void write(Compressor compressor, long crc, byte attributes, long timestamp, byte[] key, byte[] value, int valueOffset, int valueSize) {
    // write crc
    compressor.putInt((int) (crc & 0xffffffffL));
    // write magic value
    compressor.putByte(CURRENT_MAGIC_VALUE);
    // write attributes
    compressor.putByte(attributes);
    // write timestamp
    compressor.putLong(timestamp);
    // write the key
    if (key == null) {
        compressor.putInt(-1);
    } else {
        compressor.putInt(key.length);
        compressor.put(key, 0, key.length);
    }
    // write the value
    if (value == null) {
        compressor.putInt(-1);
    } else {
        int size = valueSize >= 0 ? valueSize : (value.length - valueOffset);
        compressor.putInt(size);
        compressor.put(value, valueOffset, size);
    }
}

write方法中将这些信息一一写入。CURRENT_MAGIC_VALUE代表的是当前日志的版本。Kafka不同版本中消息的格式不一样。

numRecords字段表示当前添加到该RecordBatch中的消息个数。writtenUncompressed字段表示写入的消息未压缩的总大小。因为size参数就是byte数组长度与描述信息长度的和，这些是还没有压缩时算出来的。而Records.LOG_OVERHEAD表示的是一开始写入offset和size所占的大小，显然是一个固定的值。

压缩的消息

如果指定了压缩类型，消息的格式会有所不同。

上文看到在Compressor初始化时，有这样的判断：

1
2
3
4
5

if (type != CompressionType.NONE) {
    // for compressed records, leave space for the header and the shallow message metadata
    // and move the starting position to the value payload offset
    buffer.position(initPos + Records.LOG_OVERHEAD + Record.RECORD_OVERHEAD);
}

其中LOG_OVERHEAD上面也看到过：

1
2
3

int SIZE_LENGTH = 4;
int OFFSET_LENGTH = 8;
int LOG_OVERHEAD = SIZE_LENGTH + OFFSET_LENGTH;

而RECORD_OVERHEAD为：

1

public static final int RECORD_OVERHEAD = HEADER_SIZE + TIMESTAMP_LENGTH + KEY_SIZE_LENGTH + VALUE_SIZE_LENGTH;

其中HEADER_SIZE为：

1

public static final int HEADER_SIZE = CRC_LENGTH + MAGIC_LENGTH + ATTRIBUTE_LENGTH;

因此RECORD_OVERHEAD等价于crc、magic_value、attribute、timestamp、key_size、value_size的长度加在一起。

那么，buffer.position的含义就是先预留这部分空间，之后在某时刻再把内容填充进来。

整个写的过程与上文没有差别，只是Compressor的实现不同。每条消息的格式不变。

根据上文的分析，已经知道每写入一条消息，就会给numRecords字段加1，由此可以知道最有一条消息的offset，即numRecords-1。还可以直接计算出所有消息的大小，加上RECORD-OVERHEAD的内容，就是整体的size。后面就是具体描述信息的填入，不赘述。

整个RecordBatch如下图所示：

可以将其看作嵌套的结构，外层消息包裹着内层消息。外层的“消息”的格式与内层真正的消息格式完全一致。

这样的设计是有道理的。因为消息发送到服务端后，服务端根本无法提前知道消息是否有经过压缩。由于上述描述信息的位置都是确定的，所以无论消息有没有经过压缩，服务端都会找到attributes的位置进行解析，查看压缩类型。如果是NONE，说明没有压缩，那么直接遍历即可。如果不是NONE，说明这是外层“消息”，内层还需要遍历解析，因此会找到外层value的部分，对内层真正的消息做深层遍历。
