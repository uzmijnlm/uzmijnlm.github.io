在前文中我们已经大致了解了Spring初始化的流程（主要针对AnnotationConfigApplicationContext）。主要流程分为3步：

1
2
3

AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext();
ac.register(AppConfig.class);
ac.refresh();

第一行代码核心逻辑可以描述为：对bean工厂做了初始化，添加了一些内置的bd。

第二行代码核心逻辑可以描述为：注册了一个配置类到bean工厂中。

第三行代码所做的事情非常多，在上文中粗略地介绍了其中每一个方法的作用。其中invokeBeanFactoryPostProcessors方法尤其重要，它利用内置的ConfigurationClassPostProcessor对配置类做了处理。在上文中，我们知道它会去扫描配置类指定要扫描的包以及解析配置文件等，把需要注册的bd注册到工厂中。本文将会详细分析解析的过程。

for (String beanName : candidateNames) {
   BeanDefinition beanDef = registry.getBeanDefinition(beanName);
   if (beanDef.getAttribute(ConfigurationClassUtils.CONFIGURATION_CLASS_ATTRIBUTE) != null) {

candidateNames表示的是当前工厂中所有的bd。遍历到ConfigurationClassPostProcessor时，会进入到else-if分支，进而进入checkConfigurationClassCandidate方法，在这个方法中，就会取出metadata字段进行判断：

10

Map<String, Object> config = metadata.getAnnotationAttributes(Configuration.class.getName());
if (config != null && !Boolean.FALSE.equals(config.get("proxyBeanMethods"))) {
   beanDef.setAttribute(CONFIGURATION_CLASS_ATTRIBUTE, CONFIGURATION_CLASS_FULL);
}
else if (config != null || isConfigurationCandidate(metadata)) {
   beanDef.setAttribute(CONFIGURATION_CLASS_ATTRIBUTE, CONFIGURATION_CLASS_LITE);
}
else {
   return false;
}

如果这个配置类加了@Configuration注解，则config不为null，会给bd的CONFIGURATION_CLASS_ATTRIBUTE属性设置为CONFIGURATION_CLASS_FULL；如果没有加@Configuration注解，那么会进入isConfigurationCandidate方法做判断，如果加了@Component、@ComponentScan、@Import、@ImportResource注解，或者有方法上有@Bean注解，则会给CONFIGURATION_CLASS_ATTRIBUTE属性设置为CONFIGURATION_CLASS_LITE。如果以上条件都不满足，则返回false，表示该类不是配置类。

因此，从checkConfigurationClassCandidate整体上来看，@Configuration注解并不是配置类的必要条件。只要调用register方法，就会生成一个AnnotatedGenericBeanDefinition对象，进入checkConfigurationClassCandidate方法就可以继续往下走，进行上述判断。

这里要注意的是判断注解类的条件，以及不同条件下CONFIGURATION_CLASS_ATTRIBUTE属性的值，这个值在之后会有很大影响。

protected final SourceClass doProcessConfigurationClass(
      ConfigurationClass configClass, SourceClass sourceClass, Predicate<String> filter)
      throws IOException {

   // 1.递归地处理其中包含的成员类。
   if (configClass.getMetadata().isAnnotated(Component.class.getName())) {
      processMemberClasses(configClass, sourceClass, filter);
   }

   // 2.处理@PropertySource注解。
   for (AnnotationAttributes propertySource : AnnotationConfigUtils.attributesForRepeatable(
         sourceClass.getMetadata(), PropertySources.class,
         org.springframework.context.annotation.PropertySource.class)) {
      if (this.environment instanceof ConfigurableEnvironment) {
         processPropertySource(propertySource);
      }
      else {
         logger.info("Ignoring @PropertySource annotation on [" + sourceClass.getMetadata().getClassName() +
               "]. Reason: Environment must implement ConfigurableEnvironment");
      }
   }

   // 3.处理@ComponentScan注解。
   Set<AnnotationAttributes> componentScans = AnnotationConfigUtils.attributesForRepeatable(
         sourceClass.getMetadata(), ComponentScans.class, ComponentScan.class);
   if (!componentScans.isEmpty() &&
         !this.conditionEvaluator.shouldSkip(sourceClass.getMetadata(), ConfigurationPhase.REGISTER_BEAN)) {
      for (AnnotationAttributes componentScan : componentScans) {
         // The config class is annotated with @ComponentScan -> perform the scan immediately
         Set<BeanDefinitionHolder> scannedBeanDefinitions =
               this.componentScanParser.parse(componentScan, sourceClass.getMetadata().getClassName());
         // Check the set of scanned definitions for any further config classes and parse recursively if needed
         for (BeanDefinitionHolder holder : scannedBeanDefinitions) {
            BeanDefinition bdCand = holder.getBeanDefinition().getOriginatingBeanDefinition();
            if (bdCand == null) {
               bdCand = holder.getBeanDefinition();
            }
            if (ConfigurationClassUtils.checkConfigurationClassCandidate(bdCand, this.metadataReaderFactory)) {
               parse(bdCand.getBeanClassName(), holder.getBeanName());
            }
         }
      }
   }

   // 4.处理@Import注解。
   processImports(configClass, sourceClass, getImports(sourceClass), filter, true);

   // 5.处理@ImportResource注解。
   AnnotationAttributes importResource =
         AnnotationConfigUtils.attributesFor(sourceClass.getMetadata(), ImportResource.class);
   if (importResource != null) {
      String[] resources = importResource.getStringArray("locations");
      Class<? extends BeanDefinitionReader> readerClass = importResource.getClass("reader");
      for (String resource : resources) {
         String resolvedResource = this.environment.resolveRequiredPlaceholders(resource);
         configClass.addImportedResource(resolvedResource, readerClass);
      }
   }

   // 6.处理含有@Bean的方法。
   Set<MethodMetadata> beanMethods = retrieveBeanMethodMetadata(sourceClass);
   for (MethodMetadata methodMetadata : beanMethods) {
      configClass.addBeanMethod(new BeanMethod(methodMetadata, configClass));
   }

   // 7.处理接口。
   processInterfaces(configClass, sourceClass);

   // 8.处理父类。
   if (sourceClass.getMetadata().hasSuperClass()) {
      String superclass = sourceClass.getMetadata().getSuperClassName();
      if (superclass != null && !superclass.startsWith("java") &&
            !this.knownSuperclasses.containsKey(superclass)) {
         this.knownSuperclasses.put(superclass, configClass);
         // Superclass found, return its annotation metadata and recurse
         return sourceClass.getSuperClass();
      }
   }

   return null;
}

第一步会判断配置类其中包含的成员类是否也为配置类，判断标准与之前一样。如果有的话就递归地处理。

第二步是处理@PropertySource注解，可以将注解中指定的properties文件中的内容加载到环境中，供后续使用。

第三步是扫描出@ComponentScan注解中指定的包中的所有包含@Component注解的类，注册到工厂中并完成一些初始化赋值，然后又调用parse方法递归地解析。这些类会被放到configurationClasses集合中。

第四步处理@Import注解。会递归地解析这些类，放入configurationClasses中，但不会创建bd。

第五步处理@ImportResource注解。会把注解中指定的路径添加到importedResources字段中，但不会去解析这些文件。importedResources字段属于ConfigurationClass类。

第六步处理配置类中加了@Bean的方法。会将方法封装为BeanMethod对象，添加到beanMethods字段中，但不会执行这些方法。beanMethods字段也属于ConfigurationClass类。

最后两步是处理接口和父类。

总而言之，每处理一个注解，如果发现有类需要处理，就会递归地处理。除了处理@ComponentScan时会构造bd并注册，其余步骤都是将信息添加到parser或configClass对象中，用于接下来的处理。

      ConfigurationClass configClass, TrackedConditionEvaluator trackedConditionEvaluator) {

   // 1.如果该类是通过@Import注解添加的，则构造bd并注册。
   if (configClass.isImported()) {
      registerBeanDefinitionForImportedConfigurationClass(configClass);
   }
   // 2.依次处理类中添加了@Bean注解的方法。根据方法名构造一个bd，进行初始化并注册。不会执行方法。
   for (BeanMethod beanMethod : configClass.getBeanMethods()) {
      loadBeanDefinitionsForBeanMethod(beanMethod);
   }

   // 3.解析@ImportResource注解中指定的文件，将文件中定义的bd注册到工厂中。
   loadBeanDefinitionsFromImportedResources(configClass.getImportedResources());
   // 4.处理@Import注解中实现了ImportBeanDefinitionRegistrar接口的类。
   loadBeanDefinitionsFromRegistrars(configClass.getImportBeanDefinitionRegistrars());
}

总结

本文分析了ConfigurationClassPostProcessor类如何解析一个配置类。

在refresh方法内部，会进入到PostProcessorRegistrationDelegate.invokeBeanFactoryPostProcessors方法。该方法会找到所有实现了BeanDefinitionRegistryPostProcessor接口的类，即ConfigurationClassPostProcessor类，调用其postProcessBeanDefinitionRegistry。这个方法完成了配置类的所有解析工作，将所有该注册的bd都注册到了工厂中。因为在这个过程中添加了新类，所以接下来会循环处理这个过程中产生的实现了BeanDefinitionRegistryPostProcessor接口的类。

上述逻辑执行完成后，遍历所有实现了BeanFactoryPostProcessor接口的类（当然也包括ConfigurationClassPostProcessor类，因为BeanDefinitionRegistryPostProcessor接口继承了BeanFactoryPostProcessor接口），调用其postProcessBeanFactory方法，对已经注册的bd做一些处理。这将是下文要分析的内容。
