determineConstructorsFromBeanPostProcessors方法会选出合适的构造方法，然后根据条件判断执行autowireConstructor进行实例化或者instantiateBean进行实例化。

instantiateBean方法会选择默认无参构造器进行实例化。因此这段逻辑的关键在于determineConstructorsFromBeanPostProcessors方法会在何时返回哪些构造方法，并且autowireConstructor方法会用哪个构造方法来完成实例化。本文将determineConstructorsFromBeanPostProcessors的推断过程称为第一次推断，autowireConstructor的推断过程称为第二次推断。

protected Constructor<?>[] determineConstructorsFromBeanPostProcessors(@Nullable Class<?> beanClass, String beanName)
      throws BeansException {

   if (beanClass != null && hasInstantiationAwareBeanPostProcessors()) {
      for (BeanPostProcessor bp : getBeanPostProcessors()) {
         if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            SmartInstantiationAwareBeanPostProcessor ibp = (SmartInstantiationAwareBeanPostProcessor) bp;
            Constructor<?>[] ctors = ibp.determineCandidateConstructors(beanClass, beanName);
            if (ctors != null) {
               return ctors;
            }
         }
      }
   }
   return null;
}

主要逻辑分为三部分，已由上面注释标出。

第一步是获取所有构造方法。

第二步是遍历每一个构造方法，并在不同条件下给nonSyntheticConstructors、requiredConstructor、defaultConstructor等变量赋值。

第三步是根据上面几个变量的值给candidateConstructors赋值。

可以看到条件比较多，就不一一列出来了。本文仅仅对其中出现的变量和方法做个简单的介绍。

nonSyntheticConstructors表示非合成构造器的个数。synthetic是Java中一个概念，大概是指当一个类中有一个内部类时，这个外部类可以访问内部类的private成员，这在语义上没问题，但编译器却比较难处理这个问题，所以为了访问方便，在编译期会生成一些合成的属性、方法等。这里对这个概念不深究。一般情况下我们的构造器都是非合成的。

requiredConstructor用来存放那些添加了@Autowire注解且required为true的构造方法。defaultConstructor用来存放默认构造方法。primaryConstructor与Kotlin有关，可以无视。

总的来说，选择策略可以总结如下：

  • 没有提供构造方法时，会找到一个默认无参构造器，返回null。
  • 显式提供一个默认无参构造器与上面等价，返回null。
  • 提供多个构造器时返回null。
  • 提供一个带参数的构造器时，返回该构造器。
  • 提供一个或多个构造器，只有一个构造器上有@Autowire注解（无论required为true还是false），返回该构造方法。
  • 提供多个构造器，多个构造器上有@Autowire注解且至少一个required为true，则抛异常。
  • 提供多个构造器，多个构造器上有@Autowire注解且全部为false，则这些加了注解的构造器都会返回。

执行完determineConstructorsFromBeanPostProcessors方法后，如果有构造器返回，就会进入到autowireConstructor方法进行进一步的推断。不过这里的判断条件比较多样，如果返回值为null，但该bd指定了AUTOWIRE_CONSTRUCTOR，那么也会进入到autowireConstructor方法中。
