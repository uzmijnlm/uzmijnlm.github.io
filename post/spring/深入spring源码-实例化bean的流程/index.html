前面已经分析完了bd的注册、配置类的解析以及配置类的@Configuration注解的作用等内容，上述事情做完后，剩下最重要的事情就是根据bd生成bean对象。

在前面几篇文章中，有两处已经涉及到了bean的实例化。一处是在invokeBeanFactoryPostProcessors方法中，需要用ConfigurationClassPostProcessor类去解析配置类，这个类就是在该方法中通过getBean方法生成的。其他的BeanFactoryPostProcessor也是同理。另一处就是讲到增强配置类时，里面添加了@Bean注解的方法的调用，也是在实例化bean的过程中调用的。

本文将粗略地梳理bean的实例化流程。中间涉及到的aop、属性注入、构造方法推断等逻辑将会在其他文章中详细分析。

这里存在一个很容易让人疑惑的点。按照一般的理解，这个过程要做的事情应该是实例化bean，因此在初始化过程全部结束后，我们才可以在业务代码中获得bean对象。我们都知道在业务代码里获取bean对象的方法就是getBean，通过简单的分析可以看到该方法调用的就是上面这个getBean方法。也就是说，实例化bean的过程与获取实例化后的bean的过程，调用的是同一个方法，直观上感觉这是很不合理的。Spring这样设计的其中一个重要原因是，bean对象之间可能存在循环依赖，如下：

protected Object getSingleton(String beanName, boolean allowEarlyReference) {
   Object singletonObject = this.singletonObjects.get(beanName);
   if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
      synchronized (this.singletonObjects) {
         singletonObject = this.earlySingletonObjects.get(beanName);
         if (singletonObject == null && allowEarlyReference) {
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
               singletonObject = singletonFactory.getObject();
               this.earlySingletonObjects.put(beanName, singletonObject);
               this.singletonFactories.remove(beanName);
            }
         }
      }
   }
   return singletonObject;
}

它就是从singletonObjects中拿bean对象。从狭义上讲，Spring容器指的就是这个singletonObjects。所有实例化的bean都会放在这个对象中，在业务代码里面获取一个bean对象也会经过上面的代码，最后来到这里，由于彼时所有bean都完成了实例化，所以直接就可以从singletonObjects中获取。另外，严格地说这个singletonObjects中保存的那个对象并不一定是最终我们想要得到的那个返回对象，而是要经过一些转换，不过为了叙述方便，本文统一称之为bean对象。

因为前面我们已经实例化过了Bean工厂后置处理器，所以这几个bean对象可以直接返回。而其他对象则需要进入getSingleton方法的if分支。稍后回来分析这个分支的逻辑，下面先分析那些之前已经实例化过的bean对象。

      if (singletonObject == null) {

protected void addSingletonFactory(String beanName, ObjectFactory<?> singletonFactory) {
   Assert.notNull(singletonFactory, "Singleton factory must not be null");
   synchronized (this.singletonObjects) {

protected Object getSingleton(String beanName, boolean allowEarlyReference) {
   Object singletonObject = this.singletonObjects.get(beanName);
   if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
      synchronized (this.singletonObjects) {
         singletonObject = this.earlySingletonObjects.get(beanName);
         if (singletonObject == null && allowEarlyReference) {
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
               // 从singletonFactories取出singletonFactory调用其getObject方法。
               singletonObject = singletonFactory.getObject();
               this.earlySingletonObjects.put(beanName, singletonObject);
               this.singletonFactories.remove(beanName);
            }
         }
      }
   }
   return singletonObject;
}

从getSingleton方法本身可以得知，是先从singletonObjects中获取对象；如果没有的话，并且isSingletonCurrentlyInCreation判断为true，就从earlySingletonObjects中获取对象；如果还是没有，并且allowEarlyReference为true，则从singletonFactories中获取bean对应的工厂对象，调用其getObject方法创建bean并返回。

public PropertyValues postProcessProperties(PropertyValues pvs, Object bean, String beanName) {
   InjectionMetadata metadata = findAutowiringMetadata(beanName, bean.getClass(), pvs);
   try {
      metadata.inject(bean, beanName, pvs);
   }
   catch (BeanCreationException ex) {
      throw ex;
   }
   catch (Throwable ex) {
      throw new BeanCreationException(beanName, "Injection of autowired dependencies failed", ex);
   }
   return pvs;
}


public void inject(Object target, @Nullable String beanName, @Nullable PropertyValues pvs) throws Throwable {
   Collection<InjectedElement> checkedElements = this.checkedElements;
   Collection<InjectedElement> elementsToIterate =
         (checkedElements != null ? checkedElements : this.injectedElements);
   if (!elementsToIterate.isEmpty()) {
      for (InjectedElement element : elementsToIterate) {
         if (logger.isTraceEnabled()) {
            logger.trace("Processing injected element of bean '" + beanName + "': " + element);
         }
         element.inject(target, beanName, pvs);
      }
   }
}

element.inject方法内部非常复杂，调用链也很长。简单地讲，就是解析出这个bean需要给哪几个字段赋值，然后分别对这些字段调用getBean方法获取bean对象，最后通过反射的方式赋值。但在这个过程中的getBean方法，与一个普通的没有循环依赖的bean的getBean过程有很大差别。

protected Object getSingleton(String beanName, boolean allowEarlyReference) {
   Object singletonObject = this.singletonObjects.get(beanName);
   if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
      synchronized (this.singletonObjects) {
         singletonObject = this.earlySingletonObjects.get(beanName);
         if (singletonObject == null && allowEarlyReference) {
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
               singletonObject = singletonFactory.getObject();
               this.earlySingletonObjects.put(beanName, singletonObject);
               this.singletonFactories.remove(beanName);
            }
         }
      }
   }
   return singletonObject;
}

protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
   Object exposedObject = bean;
   if (!mbd.isSynthetic() && hasInstantiationAwareBeanPostProcessors()) {
      for (BeanPostProcessor bp : getBeanPostProcessors()) {
         if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            SmartInstantiationAwareBeanPostProcessor ibp = (SmartInstantiationAwareBeanPostProcessor) bp;
            exposedObject = ibp.getEarlyBeanReference(exposedObject, beanName);
         }
      }
   }
   return exposedObject;
}

第三个参数就是早先生成的那个bean，所以这里的操作就是对该bean对象利用后置处理器做了一些处理然后返回。因为要保证此刻注入的字段是一个完整的bean（即已经经过所有后置处理器处理过的bean），因此会在这里就用这些后置处理器完成这些操作。

返回这个bean对象后，将其放入earlySingletonObjects中，并从singletonFactories中将其移除。

返回bean对象后利用反射就完成了B对象中a字段的注入。但是此时A对象中b字段的值还是为null。那么什么时候会给这个字段赋值呢？其实很简单，B的bean对象返回后，利用反射给A对象的b字段注入时，就完成了赋值。

总而言之，循环依赖的字段注入过程看起来非常复杂，但其实核心思想很简单。每当要实例化一个bean时，会先生成一个bean对象放入缓存，在对其字段赋值时，如果需要构造bean对象，则对该bean进行构造，也放入缓存中。当后面这个bean对象又引用了前一个bean时，则从缓存中将该对象取出赋值即可。后面这个对象实例化结束后，返回回来再完成前一个bean的字段赋值操作，就解决了循环依赖的问题。
