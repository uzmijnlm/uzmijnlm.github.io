整体架构

我们通常将Tomcat视为一个Servlet容器。不过从架构和源码的意义上来说，它不仅仅是一个用于装载Servlet的容器，而是一个多层级的容器结构。

Tomcat的配置文件server.xml清晰地阐明了层次关系，这里给出一个示例：

    <Context docBase="servlet" path="/servlet" reloadable="true" source="org.eclipse.jst.jee.server:servlet"/>
  </Host>
</Engine>

Engine表示Servlet引擎，引擎内部可以有多个Host，Host表示一个虚拟主机。一个Host内部可以有多个Context，一个Context表示一个应用。到了这一层就不需要在servlet.xml文件中配置了。servlet的映射在web.xml文件中有配置，或者可以通过注解等方式表示映射关系。一个Context中还包含多个Wrapper，Wrapper中包含多个Servlet。之所以还需要Wrapper这一层概念，是因为同一个Servlet类可能会有多个实例，可以用Wrapper来表示同一个类的Servlet。

这些概念在源码中都有接口一一对应。

这些接口都有相应的实现类。而这些接口又继承了更上层的接口——Container，Container又继承了LifeCycle。

此外，Tomcat中用Server类表示整个容器，在server.xml文件中，这个标签也是最外层的。用Service表示整个服务。Service中不仅包含Engine，还包含Connector，Connector用于接收不同协议的请求，然后让Engine来处理。这些都可以在server.xml文件中找到。

总的来说，整体架构如下图所示：

而启动流程，基本上可以认为是从Server开始，依次对Service、Engine以及后面的容器从上至下完成初始化（并不准确，但大致可以这样认为）。

对于一个请求来说，先经过Connector解析成请求对象。Connector中有Endpoint，表示不同的IO模型。接着将数据传递给容器的部分，从Engine开始，一层层往下传。每一层都有一个Pipeline，每个Pipeline中有多个Valve，可以自定义Valve对请求做一些处理。到了Wrapper这一层的Pipeline后会进入到FilterChain，可以自定义过滤器，最后会进入到Servlet的service方法。

protected void initInternal() throws LifecycleException {

    super.initInternal();

    if (engine != null) {
        engine.init();
    }

    // Initialize any Executors
    for (Executor executor : findExecutors()) {
        if (executor instanceof JmxEnabled) {
            ((JmxEnabled) executor).setDomain(getDomain());
        }
        executor.init();
    }

    // Initialize mapper listener
    mapperListener.init();

    // Initialize our defined Connectors
    synchronized (connectorsLock) {
        for (Connector connector : connectors) {
            try {
                connector.init();
            } catch (Exception e) {
                String message = sm.getString(
                        "standardService.connector.initFailed", connector);
                log.error(message, e);

                if (Boolean.getBoolean("org.apache.catalina.startup.EXIT_ON_INIT_FAILURE"))
                    throw new LifecycleException(message);
            }
        }
    }
}

这里对engine、executor、mapperListener、connector完成了初始化。各自的init方法并没有太复杂的逻辑。

总而言之，Catalina的load方法利用digester对许多对象完成了实例化，并在init方法中对上面这些对象完成了初始化。

protected void startInternal() throws LifecycleException {

    if(log.isInfoEnabled())
        log.info(sm.getString("standardService.start.name", this.name));
    setState(LifecycleState.STARTING);

    // Start our defined Container first
    if (engine != null) {
        synchronized (engine) {
            engine.start();
        }
    }

    synchronized (executors) {
        for (Executor executor: executors) {
            executor.start();
        }
    }

    mapperListener.start();

    // Start our defined Connectors second
    synchronized (connectorsLock) {
        for (Connector connector: connectors) {
            try {
                // If it has already failed, don't try and start it
                if (connector.getState() != LifecycleState.FAILED) {
                    connector.start();
                }
            } catch (Exception e) {
                log.error(sm.getString(
                        "standardService.connector.startFailed",
                        connector), e);
            }
        }
    }
}

总结

在启动流程中，主要是从上至下初始化并启动了多个线程准备接收数据。其中，Server、Service、Connector、Endpoint等组件完成了初始化，容器部分有Engine、Host、Context完成了初始化。在容器启动的过程中，也完成了Pipeline和Valve的初始化工作。
