在I/O模型中，有两对概念经常让人混淆——阻塞和非阻塞、同步和异步。

在不同的语境下，这些概念有着不同的内涵。其中一种比较严谨的描述来自UNIX的五种I/O模型。本文不展开讨论各种I/O模型的工作过程，仅仅对阻塞/非阻塞和同步/异步的概念做一个总结：

同步和异步IO的概念：

同步是用户线程发起I/O请求后需要等待或者轮询内核I/O操作完成后才能继续执行

异步是用户线程发起I/O请求后仍需要继续执行，当内核I/O操作完成后会通知用户线程，或者调用用户线程注册的回调函数

阻塞和非阻塞IO的概念：

阻塞是指I/O操作需要彻底完成后才能返回用户空间

非阻塞是指I/O操作被调用后立即返回一个状态值，无需等I/O操作彻底完成

具体的内涵需要理解系统调用函数和上面几种I/O模型对它们的应用才能真正理解。

本文再提供一种描述，仅仅作为参考：

同步与异步的区别：是否使用回调函数

阻塞与非阻塞的区别：是否使用多线程

这种理解非常的不严谨，但也有一定的道理。因为异步操作是基于“通知”，所以一般会定义一个回调函数；而使用多线程自然可以避免线程的等待，实现非阻塞。

由于本文不涉及I/O模型的讨论，所以将模糊处理这两对概念的定义。下面说到“异步”，仅表示开启另一个线程去执行一个任务，即与主线程不是同步。

public class SyncExample {
    public static void doSomethingA() {
        System.out.println("--- doSomethingA ---");
    }

    public static void doSomethingB() {
        System.out.println("--- doSomethingB ---");
    }

这里定义了两个任务，即两个方法doSomethingA和doSomethingB。在main方法中逐个执行。这是最简单的同步任务。

public class AsyncExample {
    public static void doSomethingA() {
        System.out.println("--- doSomethingA ---");
    }

    public static void doSomethingB() {
        System.out.println("--- doSomethingB ---");
    }

    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(AsyncExample::doSomethingA);
        thread.start();
        doSomethingB();
    }
}

这样做的确很简单，但并不适用于实际场景，主要原因有亮点：

1.实际场景中任务个数非常多，如果每次都创建一个新的线程则资源消耗太大。

2.获取一个线程任务运行结束后的结果比较麻烦。

第一个问题可以直接用线程池来解决。第二个问题有多种解决方式，其中一种是用共享变量，但是这要求变量定义在外部，而不是属于thread对象本身；另一种方式基于JDK的Future。本文深入探讨第一个问题。

public class ThreadPoolExample {
    private final static int AVAILABLE_PROCESSORS = Runtime.getRuntime().availableProcessors();
    private final static ThreadPoolExecutor POOL_EXECUTOR = new ThreadPoolExecutor(
            AVAILABLE_PROCESSORS,
            AVAILABLE_PROCESSORS * 2,
            1,
            TimeUnit.MINUTES,
            new LinkedBlockingDeque<>(5),
            new CallerRunsPolicy());

    public static void doSomethingA() {
        System.out.println("--- doSomethingA ---");
    }

    public static void doSomethingB() {
        System.out.println("--- doSomethingB ---");
    }

    public static void main(String[] args) {
        POOL_EXECUTOR.execute(ThreadPoolExample::doSomethingA);
        POOL_EXECUTOR.execute(ThreadPoolExample::doSomethingB);
    }
}

public ThreadPoolExecutor(int corePoolSize,
                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler) {
    if (corePoolSize < 0 ||
        maximumPoolSize <= 0 ||
        maximumPoolSize < corePoolSize ||
        keepAliveTime < 0)
        throw new IllegalArgumentException();
    if (workQueue == null || threadFactory == null || handler == null)
        throw new NullPointerException();
    this.acc = System.getSecurityManager() == null ?
            null :
            AccessController.getContext();
    this.corePoolSize = corePoolSize;
    this.maximumPoolSize = maximumPoolSize;
    this.workQueue = workQueue;
    this.keepAliveTime = unit.toNanos(keepAliveTime);
    this.threadFactory = threadFactory;
    this.handler = handler;
}

corePoolSize表示核心线程数，maximumPoolSize表示池中允许的最大线程数。从这两个参数可以推测出线程池中的线程个数是一个动态的变量，可以超过核心线程数，但不能超过最大线程数。keepAliveTime表示非核心不活跃线程最大存活时间。从这个参数可以看出线程池中的线程应该有不同的状态，至少可以分为活跃和不活跃。unit表示时间单位。workQueue表示阻塞队列类型，这个队列中的元素类型是Runnable，显然是存放任务的队列。threadFactory对象用来创建新的线程。handler定义了拒绝一个任务时的处理逻辑。

private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0));
private static final int COUNT_BITS = Integer.SIZE - 3;
private static final int CAPACITY   = (1 << COUNT_BITS) - 1;

// runState is stored in the high-order bits
private static final int RUNNING    = -1 << COUNT_BITS;
private static final int SHUTDOWN   =  0 << COUNT_BITS;
private static final int STOP       =  1 << COUNT_BITS;
private static final int TIDYING    =  2 << COUNT_BITS;
private static final int TERMINATED =  3 << COUNT_BITS;

private static int ctlOf(int rs, int wc) { return rs | wc; }

字段ctl用来同时表示线程池的状态（高3位）和线程个数（低29位）。计算这个值用到了ctlOf方法，其中又用到了RUNNING字段。RUNNING表示线程池的状态。线程池一共有上面代码中的五种状态，通过COUNT_BITS算出来。COUNT_BITS的值就是29，即通过一个数左移29位得到状态值。那么几种状态值分别如下：

RUNNING（接收新任务并处理阻塞队列里的任务）: 111 000……

SHUTDOWN（拒绝新任务但是处理阻塞队列里的任务）: 000 000……

STOP（拒绝新任务并且抛弃阻塞队列里的任务，同时中断正在处理的任务）: 001 000……

TIDYING（所有任务都执行完，活跃线程为0，将要调用terminated方法）: 010 000……

TERMINATED（终止状态，即terminated方法调用完以后的状态）: 011 000……

至于这几种状态怎么转换，在之后的代码分析中如果涉及到会再进行分析。

    }
    else if (!addWorker(command, false)) //6. 如果队列满了，则新增线程，新增失败则执行拒绝策略。
        reject(command);

private boolean addWorker(Runnable firstTask, boolean core) {
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        // Check if queue empty only if necessary.
        if (rs >= SHUTDOWN &&
            ! (rs == SHUTDOWN &&
               firstTask == null &&
               ! workQueue.isEmpty()))
            return false;

        for (;;) {
            int wc = workerCountOf(c);
            if (wc >= CAPACITY ||
                wc >= (core ? corePoolSize : maximumPoolSize))
                return false;
            if (compareAndIncrementWorkerCount(c))
                break retry;
            c = ctl.get();  // Re-read ctl
            if (runStateOf(c) != rs)
                continue retry;
            // else CAS failed due to workerCount change; retry inner loop
        }
    }

    boolean workerStarted = false;
    boolean workerAdded = false;
    Worker w = null;
    try {
        w = new Worker(firstTask);
        final Thread t = w.thread;
        if (t != null) {
            final ReentrantLock mainLock = this.mainLock;
            mainLock.lock();
            try {
                // Recheck while holding lock.
                // Back out on ThreadFactory failure or if
                // shut down before lock acquired.
                int rs = runStateOf(ctl.get());

                if (rs < SHUTDOWN ||
                    (rs == SHUTDOWN && firstTask == null)) {
                    if (t.isAlive()) // precheck that t is startable
                        throw new IllegalThreadStateException();
                    workers.add(w);
                    int s = workers.size();
                    if (s > largestPoolSize)
                        largestPoolSize = s;
                    workerAdded = true;
                }
            } finally {
                mainLock.unlock();
            }
            if (workerAdded) {
                t.start();
                workerStarted = true;
            }
        }
    } finally {
        if (! workerStarted)
            addWorkerFailed(w);
    }
    return workerStarted;
}

线程的关闭和任务的拒绝策略

线程池有shutdown和shutdownNow两种方式关闭。调用shutdown后，线程池就不会再接收新的任务，但是工作队列里的任务还是要执行。调用shutdownNow后，工作队列中的任务也会丢弃，正在执行的任务会中断。其中涉及到线程池状态的转换，本文不展开讨论。

而在execute方法中我们看到符合某些条件时会执行拒绝策略。其中默认的拒绝策略如下：

1
2
3
4
5

public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    if (!e.isShutdown()) {
        r.run();
    }
}

它会直接执行线程的run方法。如果线程池已经关闭，则会丢弃该任务。

总结

使用线程可以实现异步的效果，但是创建和销毁线程非常消耗资源，所以实际场景中常用线程池来实现线程的复用。线程池的基本原理是，添加一个任务时，正常情况下直接创建线程执行；如果线程数大于等于线程池中的核心线程数，则将任务放入队列，依次由之前创建的线程来执行。
