Java基础 - 基于JDK中的Future实现异步编程

上文讲到说用用线程实现异步编程，有两个问题比较突出：

1.实际场景中任务个数非常多，如果每次都创建一个新的线程则资源消耗太大。

2.获取一个线程任务运行结束后的结果比较麻烦。

第一个问题可以通过线程池解决。要解决第二个问题，还需要对线程的相关类做进一步的封装。

在JDK中可以利用Future相关类解决该问题。下面看一个最简单的例子：

public class AsyncFutureExample {
    public static String doSomethingA() {
        System.out.println("--- doSomethingA ---");
        return "TaskAResult";
    }

    public static String doSomethingB() {
        System.out.println("--- doSomethingB ---");
        return "TaskBResult";
    }

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        FutureTask<String> futureTask = new FutureTask<>(AsyncFutureExample::doSomethingA);
        Thread thread = new Thread(futureTask);
        thread.start();
        String taskBResult = doSomethingB();
        String taskAResult = futureTask.get();
        System.out.println(taskAResult + " " + taskBResult);
    }
}

doSomethingA和doSomethingB方法都是有返回值的，这时就需要将它们封装在FutureTask对象中，然后把FutureTask对象封装到线程中提交。之后，如果想要获取到任务的返回值，调用get方法即可。

虽然目前还不知道FutureTask内部封装了哪些东西，也不知道为什么get方法可以返回执行结果，但至少可以推测出来，FutureTask也是一个Runnable对象，在启动了该任务后会执行我们定义的方法，而且它内部会有一些字段表示它的状态。而get方法必然是一个阻塞的方法（很可能就是一个死循环，只有当任务执行结束后，通过内部状态的改变才能结束循环），任务结束后获得返回值。

而把上述代码转换为用线程池实现也是非常简单的，直接把FutureTask对象提交给execute方法即可；或者直接通过submit方法传入执行逻辑。获取返回值的方式是一样的。

private static final sun.misc.Unsafe UNSAFE;

static {

        Class<?> k = FutureTask.class;

public void run() {
  
  ......
    
    try {
        Callable<V> c = callable;
        if (c != null && state == NEW) {
            V result;
            boolean ran;
            try {
                // 1.执行任务，得到返回值，将ran设置为true，表示任务已经跑完了。
                result = c.call();
                ran = true;
            } catch (Throwable ex) {
                // 2.如果执行任务过程中抛异常，则将返回值设为null，ran设为false，表示没跑完。
                result = null;
                ran = false;
                setException(ex);
            }
            if (ran)
                // 3.任务执行正常，设置正确的返回结果。
                set(result);
        }
    } finally {
        runner = null;
        int s = state;
        if (s >= INTERRUPTING)
            handlePossibleCancellationInterrupt(s);
    }
}

核心逻辑很简单，就是去执行之前传入的任务，然后设置返回值。下面看一下各个方法内部做了什么。首先是当任务出现异常时的setException方法：

protected void setException(Throwable t) {
    if (UNSAFE.compareAndSwapInt(this, stateOffset, NEW, COMPLETING)) {
        outcome = t;
        UNSAFE.putOrderedInt(this, stateOffset, EXCEPTIONAL); // final state
        finishCompletion();
    }
}

在这个方法中，将状态从NEW改为了COMPLETING，将异常对象设置给了outcome字段，然后设置状态为EXCEPTIONAL，最后执行finishCompletion。

private void finishCompletion() {
    for (WaitNode q; (q = waiters) != null;) {
        if (UNSAFE.compareAndSwapObject(this, waitersOffset, q, null)) {
            for (;;) {
                Thread t = q.thread;
                if (t != null) {
                    q.thread = null;
                    LockSupport.unpark(t);
                }
                WaitNode next = q.next;
                if (next == null)
                    break;
                q.next = null; 
                q = next;
            }
            break;
        }
    }

    done();

    callable = null; 
}

这个方法的核心逻辑是从头开始遍历waiters链表，唤醒（unpark）它封装的那个线程，然后把没有用的对象全部置为null，方便gc。这里涉及到了一个unpark操作，所以一定在某处有一个park操作。目前还看不太明白这个方法在做什么，需要结合get方法来看。最后的done方法是一个空实现。

private int awaitDone(boolean timed, long nanos)
    throws InterruptedException {
    final long deadline = timed ? System.nanoTime() + nanos : 0L;
    WaitNode q = null;
    boolean queued = false;
    for (;;) {
        if (Thread.interrupted()) {
            removeWaiter(q);
            throw new InterruptedException();
        }

        int s = state;
        if (s > COMPLETING) {
            // 1.状态已经终止时，将WaitNode对象的thread字段置为null，并返回当前状态。
            if (q != null)
                q.thread = null;
            return s;
        }
        else if (s == COMPLETING) 
            // 2.状态为COMPLETING时，暂停当前线程。
            Thread.yield();
        else if (q == null)
            // 3.状态还没有到COMPLETING前，并且q变量的值为null，则给它赋值。
            q = new WaitNode();
        else if (!queued)
            // 4.如果还没有放入waiters等待列表，则将其放入。
            queued = UNSAFE.compareAndSwapObject(this, waitersOffset,
                                                 q.next = waiters, q);
        else if (timed) {
            // 5.设置了超时时间的情况下的操作。
            nanos = deadline - System.nanoTime();
            if (nanos <= 0L) {
                removeWaiter(q);
                return state;
            }
            LockSupport.parkNanos(this, nanos);
        }
        else
            // 6.状态还没有到COMPLETING之前，将线程park。
            LockSupport.park(this);
    }
}

这个方法的核心逻辑就是一段死循环，与之前的推测一致。最简单的情况，自然是第一次循环时任务就已经跑完了，直接到了步骤1的地方返回状态值，不需要做后面的循环。但很多时候， 可能任务运行时间都比较长，这个for循环会执行很多次。这时，流程一般会是先判断任务是否已经终止，如果没有则创建一个WaitNode赋值给变量q，然后将其添加到waiters链表后面。然后park当前线程。等到任务完成后，会有别的线程来unpark当前线程，也就是上面的finishCompletion方法。之所以waiters是一个链表，是因为可能会有多个线程都去调用get方法。当任务没有结束的时候，需要有一个链表来维护它们。而在finishCompletion方法中，任务已经跑完，这时可以将这些线程一一唤醒。

private V report(int s) throws ExecutionException {
    Object x = outcome;
    if (s == NORMAL)
        return (V)x;
    if (s >= CANCELLED)
        throw new CancellationException();
    throw new ExecutionException((Throwable)x);
}

如果运行正常，则将结果返回。否则抛异常。

public boolean cancel(boolean mayInterruptIfRunning) {
    if (!(state == NEW &&
          UNSAFE.compareAndSwapInt(this, stateOffset, NEW,
              mayInterruptIfRunning ? INTERRUPTING : CANCELLED)))
        return false;
    try {    // in case call to interrupt throws exception
        if (mayInterruptIfRunning) {
            try {
                Thread t = runner;
                if (t != null)
                    t.interrupt();
            } finally { // final state
                UNSAFE.putOrderedInt(this, stateOffset, INTERRUPTED);
            }
        }
    } finally {
        finishCompletion();
    }
    return true;
}

这个方法主要是给任务设置状态，如果可以中断的话则中断线程。

总结

FutureTask类出现的目的，主要就是想获取一个任务运行结束后的结果。FutureTask在继承了Runnable接口的基础上，还继承了Future接口，并且FutureTask类的内部定义了字段表示状态和任务运行的结果。在实现上，就是一个线程去执行任务，另一个线程等待任务结束获取结果并返回。
