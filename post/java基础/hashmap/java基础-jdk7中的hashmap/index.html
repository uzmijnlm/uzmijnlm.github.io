众所周知，JDK中的HashMap是通过数组+链表（树）实现的，大致如下图所示：

但是在不同版本中有不同的实现。比较经典的就是JDK1.7和JDK1.8的实现。1.8相对于1.7有较大的优化，并且此后基本没有其他改动。本文先分析1.7版本的HashMap（数组+链表），这样能清晰地了解到JDK的实现思路，下面几篇文章会对JDK1.8版本的实现做分析。

public V put(K key, V value) {
    // 1.初始化bucket数组。
    if (table == EMPTY_TABLE) {
        inflateTable(threshold);
    }
    // 2.处理key为null的情况。
    if (key == null)
        return putForNullKey(value);
    // 3.处理key不为null的情况。
    int hash = hash(key);
    int i = indexFor(hash, table.length);
    for (Entry<K,V> e = table[i]; e != null; e = e.next) {
        Object k;
        if (e.hash == hash && ((k = e.key) == key || key.equals(k))) {
            V oldValue = e.value;
            e.value = value;
            e.recordAccess(this);
            return oldValue;
        }
    }

    modCount++;
    addEntry(hash, key, value, i);
    return null;
}

private void inflateTable(int toSize) {
    // 找到大于toSize的最小的二次幂数。
    int capacity = roundUpToPowerOf2(toSize);

    threshold = (int) Math.min(capacity * loadFactor, MAXIMUM_CAPACITY + 1);
    table = new Entry[capacity];
    initHashSeedAsNeeded(capacity);
}

这里调用了roundUpToPowerOf2方法去找大于toSize的最小的二次幂数。然后利用该值重新计算了threshold字段的值，接着对table进行初始化，即一个capacity长度的Entry数组。这里看一下数组中的元素Entry类：

private static int roundUpToPowerOf2(int number) {
    // assert number >= 0 : "number must be non-negative";
    return number >= MAXIMUM_CAPACITY
            ? MAXIMUM_CAPACITY
            : (number > 1) ? Integer.highestOneBit((number - 1) << 1) : 1;
}

对于一般的入参number，只需要看Integer.highestOneBit((number - 1) « 1)的值即可。

public static int highestOneBit(int i) {
    // HD, Figure 3-1
    i |= (i >>  1); // 把最高位的1复制到最高位右边一位。
    i |= (i >>  2); // 把最高位的两个1往右复制。
    i |= (i >>  4); // 把最高位的四个1往右复制。
    i |= (i >>  8); // 把最高位的八个1往右复制。
    i |= (i >> 16); // 把最高位的十六个1往右复制。
    return i - (i >>> 1); // 右移一位，用原数减去这个数，得到最高位为1，其他位全为0的数。即小于等于入参i的最大的二次幂数。
}

这个方法的逻辑已在注释中标出。由于调用这个方法时将原数number减了1并左移了一位，因此算出来就是大于number的最小的二次幂数。用number左移一位来计算是比较容易理解的，但是为什么要先减1呢？这是因为highestOneBit方法返回的是“小于等于”入参i的二次幂数，也就说，如果number为16，左移一位变为32，那么最终返回的就是32，与我们想要的结果不一致。因此先减1再左移，就可以得到比32小的数（30），最终返回16。

final int hash(Object k) {
    int h = hashSeed;
    if (0 != h && k instanceof String) {
        return sun.misc.Hashing.stringHash32((String) k);
    }

    h ^= k.hashCode();

    h ^= (h >>> 20) ^ (h >>> 12);
    return h ^ (h >>> 7) ^ (h >>> 4);
}

这里暂时不分析取值的逻辑，只需要知道这个hash值并不是直接调用对象的hashCode方法取到的。

接着根据hash值得到bucket数组的索引：

1
2
3
4

static int indexFor(int h, int length) {
    // assert Integer.bitCount(length) == 1 : "length must be a non-zero power of 2";
    return h & (length-1);
}

这里用了一个&运算。由于之前数组的长度为二次幂数，因此这里length-1就可以得到一个全为1的数，与任何数做&运算都能保证结果在数组长度范围内，并且由于hash值h的值有均匀性，所以与全为1的数做&运算后也能保证均匀性。这种设计也是提高效率的小技巧之一，因为%运算是效率比较低的。

这时回看上面的hash方法。

因为在indexFor方法中h与length-1做&运算，因此实际上只有h的低位在参与运算。为了让对象本身hash值的高位也影响到该运算，所以在hash方法中就有了很多右移运算。

void addEntry(int hash, K key, V value, int bucketIndex) {
    if ((size >= threshold) && (null != table[bucketIndex])) {
        resize(2 * table.length);
        hash = (null != key) ? hash(key) : 0;
        bucketIndex = indexFor(hash, table.length);
    }

    createEntry(hash, key, value, bucketIndex);
}


void createEntry(int hash, K key, V value, int bucketIndex) {
    Entry<K,V> e = table[bucketIndex];
    table[bucketIndex] = new Entry<>(hash, key, value, e);
    size++;
}

首先会判断是否要扩容。先来看不需要扩容的情况，直接调用createEntry方法。这是直接在bucket的链表的头部插入了新元素，size加1。如果需要扩容，则调用resize方法先扩容，然后重新计算hash值和索引，然后调用createEntry方法添加元素。

这个过程与处理key为null的过程基本一致，只是key为null时索引取为0。

void resize(int newCapacity) {
    Entry[] oldTable = table;
    int oldCapacity = oldTable.length;
    if (oldCapacity == MAXIMUM_CAPACITY) {
        threshold = Integer.MAX_VALUE;
        return;
    }

    Entry[] newTable = new Entry[newCapacity];
    transfer(newTable, initHashSeedAsNeeded(newCapacity));
    table = newTable;
    threshold = (int)Math.min(newCapacity * loadFactor, MAXIMUM_CAPACITY + 1);
}

可以看到核心逻辑是调用transfer方法，然后重新对threashold赋值。这里同样调用了initHashSeedAsNeeded方法，暂时先不用管。

void transfer(Entry[] newTable, boolean rehash) {
    int newCapacity = newTable.length;
    for (Entry<K,V> e : table) {
        while(null != e) {
            Entry<K,V> next = e.next;
            if (rehash) {
                e.hash = null == e.key ? 0 : hash(e.key);
            }
            int i = indexFor(e.hash, newCapacity);
            e.next = newTable[i];
            newTable[i] = e;
            e = next;
        }
    }
}

逻辑比较简单，就是对每一个元素重新计算它应该被放到哪里，然后对其进行放置。rehash的值就是前面initHashSeedAsNeeded的返回值，决定是否需要重新计算hash值。

final boolean initHashSeedAsNeeded(int capacity) {
    boolean currentAltHashing = hashSeed != 0;
    boolean useAltHashing = sun.misc.VM.isBooted() &&
            (capacity >= Holder.ALTERNATIVE_HASHING_THRESHOLD);
    boolean switching = currentAltHashing ^ useAltHashing;
    if (switching) {
        hashSeed = useAltHashing
            ? sun.misc.Hashing.randomHashSeed(this)
            : 0;
    }
    return switching;
}

这个方法就是根据环境变量等判断条件计算hashSeed，返回值表示在扩容时是否需要重新计算hash值。hashSeed在计算hash值时会增加散列程度。默认为0。

扩容时的多线程问题

从上面transfer方法可以看出，数组在扩容后，由于头插法会导致原链表的顺序反过来。因此，如果有多个线程同时在对这个对象扩容，如果其中一个线程先完成扩容，另一个线程才开始扩容，则会因为顺序的问题导致后一个线程扩容后形成循环链表。

那么，为什么在扩容时要重新计算每个元素的位置并用头插法重新放置，而不是直接把原来的Entry对象直接移过去呢？这是因为扩容的目的就是减小链表的长度，使数据重新均匀分布，所以要对每个元素都重新放置。

public V get(Object key) {
    if (key == null)
        return getForNullKey();
    Entry<K,V> entry = getEntry(key);

    return null == entry ? null : entry.getValue();
}


final Entry<K,V> getEntry(Object key) {
    if (size == 0) {
        return null;
    }

    int hash = (key == null) ? 0 : hash(key);
    for (Entry<K,V> e = table[indexFor(hash, table.length)];
         e != null;
         e = e.next) {
        Object k;
        if (e.hash == hash &&
            ((k = e.key) == key || (key != null && key.equals(k))))
            return e;
    }
    return null;
}


private V getForNullKey() {
    if (size == 0) {
        return null;
    }
    for (Entry<K,V> e = table[0]; e != null; e = e.next) {
        if (e.key == null)
            return e.value;
    }
    return null;
}

public V remove(Object key) {
    Entry<K,V> e = removeEntryForKey(key);
    return (e == null ? null : e.value);
}


final Entry<K,V> removeEntryForKey(Object key) {
    if (size == 0) {
        return null;
    }
    int hash = (key == null) ? 0 : hash(key);
    int i = indexFor(hash, table.length);
    Entry<K,V> prev = table[i];
    Entry<K,V> e = prev;

    while (e != null) {
        Entry<K,V> next = e.next;
        Object k;
        if (e.hash == hash &&
            ((k = e.key) == key || (key != null && key.equals(k)))) {
            modCount++;
            size--;
            if (prev == e)
                table[i] = next;
            else
                prev.next = next;
            e.recordRemoval(this);
            return e;
        }
        prev = e;
        e = next;
    }

    return e;
}

唯一需要注意的是在删除操作中对modCount字段也做了加1的操作。

private abstract class HashIterator<E> implements Iterator<E> {
    Entry<K,V> next;        // next entry to return
    int expectedModCount;   // For fast-fail
    int index;              // current slot
    Entry<K,V> current;     // current entry

    HashIterator() {
        expectedModCount = modCount;
        if (size > 0) { // advance to first entry
            Entry[] t = table;
            while (index < t.length && (next = t[index++]) == null)
                ;
        }
    }

final Entry<K,V> nextEntry() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
    Entry<K,V> e = next;
    if (e == null)
        throw new NoSuchElementException();

    if ((next = e.next) == null) {
        Entry[] t = table;
        while (index < t.length && (next = t[index++]) == null)
            ;
    }
    current = e;
    return e;
}

由于每一次remove方法都会对modCount值加1，因此第二次调用nextEntry方法时，modCount值与expectedModCount值就不相等了，由此抛出ConcurrentModificationException异常。

这里其实就是用迭代器+删除操作，揭示了多线程下如果其中一个线程对map做了写入或删除操作，对modCount的值进行了修改，则另一个线程对map进行遍历时就会发生异常。modCount就表示修改的次数。

public void remove() {
    if (current == null)
        throw new IllegalStateException();
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
    Object k = current.key;
    current = null;
    HashMap.this.removeEntryForKey(k);
    expectedModCount = modCount;
}

虽然它也调用了removeEntryForKey方法，但是最后将modCount的最新值赋给了expectedModCount。

总结

JDK1.7的HashMap的数据结构是数组+链表。对链表的插入操作是用的头插法。该HashMap是非线程安全的。如果多个线程对其进行操作，可能在扩容时造成循环链表，从而会在get元素时死循环。另外，也无法在一个线程对map进行修改时，另一个线程对其进行遍历等操作。
